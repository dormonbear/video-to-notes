## 一、Agent 架构与核心能力

### 1. PiAgent 架构：极简、透明、事件驱动
> [从外向里解读PiAgent架构](https://dormon.net/posts/2026-06-16-douyin-7648162171506986240/)

一个最小可用 Agent 框架由四个模块组成：

- **AgentLoop（代理循环）**：驱动核心。LLM 本身无状态，靠循环赋予 Agent 持续工作、自我修复、调用工具的能力。
- **Context（上下文）**：记录对话情境（system message + history）。用 Transformer 模块压缩历史防止上下文过长。Memory 用**树形结构**存储（每个节点含 ID 与 parentID），便于检索对话历史。
- **Tools（工具）**：极简原则，基础工具为 `read / write / edit / bash`，内建权限管理（如只读模式）。
- **System Message 管理**：默认消息 → `append-system.md` 文件追加 → `<skills>` 标签定义技能 → `--system` 命令行完全覆写，四级灵活定制。
- **扩展机制（Hooks）**：事件驱动。`pre_tool_call`（调用前，可 pass/deny）与 `post_tool_call`（调用后，处理结果），用于安全检查、日志、结果后处理等可插拔逻辑。

### 2. 从零开发 Agent 的 10 项核心能力
> [从零开发AI Agent所需10项技能](https://dormon.net/posts/2026-06-14-douyin-7648892907285417268/)

以 ZhTalk 项目为例，列出一个完整 Agent 所需能力栈：

| 能力 | 作用 |
|---|---|
| **ReAct 循环** | 基石：Reasoning（推理）+ Action（行动），判断是否调工具 |
| **Tools** | LLM 的"手脚眼耳鼻"：文件读写、exec、run_python、web_search/fetch |
| **Skills** | 对 Tools 的封装管理；按需 `load_skill`，避免工具过多时模型选不准 |
| **Memory** | 短期（当前会话）/ 长期（跨会话）/ 用户画像；用类 SQLite FTS 检索，含相关性与时间衰减 |
| **Context** | 无状态请求需附带历史；多层压缩（micro/tool/auto-compact）降 Token 成本 |
| **Permission** | read/write/exec/network 分级；高危拒绝、中危人在回路、低危放行 |
| **Sub-agent** | 复杂任务拆分并行，各子 Agent 独立上下文/工具，结果回主 Agent 汇总 |
| **Sessions** | 多会话管理：create / list / rewind 回溯 |
| **Command** | slash-command 交互：new / rewind / sessions / compact / system-cmd |
| **Hook** | pre-tool / post-tool / 会话前后插入自定义逻辑 |
| **Query Engine** | API 请求、流式、缓存、重试、限流、Token 限制的综合处理 |

**技术栈**：Node.js + pnpm + TypeScript + LangChain/LangGraph + Commander.js。

## 二、记忆系统：长期记忆的写入与治理
> [Agent长期记忆的写入机制与评估](https://dormon.net/posts/2026-06-16-douyin-7647441229726190883/)

长期记忆不是存全部上下文，而是**跨 Session 复用的知识层**，只存三类：**偏好（Preference）/ 事实（Fact）/ 经验（Experience）**。

**写入的五大工程关卡：**
1. **写入准则（Write Gate）**：是不是一次性信息？有无跨会话复用价值？能否归入三类之一？三问过滤。
2. **触发时机（When to Write）**：用户强信号 / 任务结束 / LLM 自评分超阈值（如 ≥0.8），不是每轮都写。
3. **抽取归一（ETL of Memory）**：Extract（实体+断言）→ Normalize（去重、冲突合并，如 `city: BJ→HZ` 决定覆盖还是并存）。
4. **衰减淘汰（Decay/TTL）**：靠 TTL 倒计时、命中次数、权重条；状态在 Active / 降权 / Evict 间流转。
5. **隐私与可控（Privacy）**：PII 与 Secret 打标隔离加密；用户可见、可改、可删（符合 GDPR）。

**可观测指标**：写入量、命中率、冲突率、用户删除率——防止"写得多用得少"的 Memory 污染。

> 核心理念：把长期记忆设计成一个**可判断、可归一、可衰减、可审计**的子系统。

## 三、可靠性：死循环与稳定性

### 1. 死循环排查：四重关卡
> [Agent死循环排查与解决：四重关卡](https://dormon.net/posts/2026-06-16-douyin-7649748416737627407/)

**三种死循环长相**：① 反复调同一动作；② 两状态间 A-B-A-B 横跳；③ 工具失败仍硬重试。

- **第一关 检测**：状态 Hash 去重 → 动作历史比对 → 步数阈值（如 100 步才开始判断）→ 相似度计算 → 报警。**看轨迹不只看单步**。
- **第二关 打破**：最大步数上限 / 检测到重复就强制换策略 / 给决策加随机性。换策略 ≠ 瞎换 = 换工具、改查询、退一步重新规划。
- **第三关 根因修复**：① Prompt 不清（目标模糊/约束不足/缺示例/输出格式不明）；② **Observation 没回灌**（工具结果没更新进上下文 → 每轮看到相同输入 → 重复同一动作，最常见）；③ 缺少停止条件。
- **第四关 兜底**：回退到上一个 Checkpoint。

> 目标：构建**可检测、可打破、可定位、有兜底**的稳定系统。

### 2. 可靠性与安全综合
> [Agent的可靠性与安全性：面面俱到](https://dormon.net/posts/2026-06-16-douyin-7649309207028816886/)

按面试七问梳理：
1. **死循环处理**：循环次数上限 + 任务超时 + 模型自判停止（提示输出 `STOP`）。
2. **输出稳定性**：强制 JSON Schema + 校验重试（如 3 次）+ 失效降级引擎 + 敏感场景 `temperature=0`。
3. **LLM 服务挂了**：兜底文案/if-else 逻辑 + 多供应商（主 Claude 备 GPT-4）+ 熔断切换。
4. **Prompt 注入防御**：输入过滤转义 + 权限最小 + 危险操作二次确认 + 用户输入与系统指令隔离。
5. **防危险操作**：只开必要工具 + 数据范围限制 + 操作权限限制 + 人工确认 + 沙箱 + 日志审计。
6. **幻觉处理**：工具结果作数据来源 + 精确数据走数据库不走 LLM + 要求引用来源 + 二次校验 Agent。
7. **Tool 调用失败**：反思换参重试 / 权限不足请求授权 / 彻底不可用则跳过并标记。

## 四、安全：高危操作防护机制
> [Agent高危操作防护机制设计与落地](https://dormon.net/posts/2026-06-16-douyin-7651279852976295174/)

当 Agent 拿到操作数据库、发邮件等高危权限时，需要一个**权限—确认—审批—回滚—追溯的闭环**：

1. **身份与租户边界**：每次工具调用绑定 `user_id / role / session_id / resource_scope`，防越权、隔离多租户数据。
2. **最小权限原则**：数据库默认只读、按表/字段授权；邮件限白名单收件人、限群发量。
3. **动作分级**：低（查询，放行）/ 中（生成草稿，简单确认）/ 高（发邮件，二次确认）/ 极高（删除、批量改库，严格审批）。
4. **数据与邮件护栏**：
   - 数据库：禁裸写 SQL，模板化 + 参数化 + 限行数（默认 100）+ 写操作走审批（记原因/影响/回滚）。
   - 邮件：正文脱敏 + 附件扫描 + 收件人校验 + 人工审核。
5. **沙箱/事务/回滚**：先展示计划与 diff 预览（Human-in-the-loop）→ 沙箱执行 → 事务 BEGIN → 影子表验证 → COMMIT；失败/拒绝即 ROLLBACK，邮件先存草稿。
6. **审计监控**：全程记录 prompt/参数/执行者/审批人/结果；外部输入只作数据不改策略；异常告警 + 防注入清洗。

## 五、工程落地：Agent 系统设计要点
> [Agent系统设计十二问](https://dormon.net/posts/2026-06-16-douyin-7638950515585387782/)

落地能力的系统化拆解（节选要点）：
1. **Agent 划分**：按职责边界（需求理解/检索/生成/审核/发布），不是按模型数量；每个有明确 I/O、独立权限、失败兜底。
2. **LangChain vs LangGraph**：前者是 LLM 应用工具箱（模型调用、Prompt、工具、检索、Chains 线性编排）；后者专注**流程编排与状态机**，处理多节点/条件分支/循环。
3. **何时用 LangGraph**：Agent 流程非线性（审核重写、工具重试、人工确认、分支走向），图结构让状态与跳转可控。
4. **节点/边/状态**：节点=任务单元；边=流程走向（普通边/条件边）；状态=全局上下文（输入、中间结果、错误次数、评分、阶段）。
5. **状态管理**：最常见问题是状态混乱。解法 = 状态分层 + 关键字段版本与来源 + 节点责任隔离（只改自己字段）。
6. **循环控制与错误恢复**：最大重试次数 + 明确退出条件 + 降级路径（模板答案/部分结果/转人工）。
7. **死循环策略**：状态记录跳转原因 + 循环节点次数上限 + 连续无提升强制退出 + 保留可用版本兜底。
8. **无效重写控制**：用**多维评分**定义"改进"（事实准确性/结构完整/语气匹配/敏感风险/长度合规）；重写前传失败原因，重写后对比分数，无提升则停止或切换策略。
> （后续 9–12 题延续内容安全、评估体系等，逻辑一致：工程化、可观测、可兜底。）

## 贯穿全站的核心思想

1. **Agent = ReAct 循环 + 工具 + 记忆 + 上下文管理**，框架（PiAgent/LangGraph）只是把这些模块化。
2. **可靠性靠"四可"**：可检测、可打破、可定位、有兜底；死循环根因常是 Observation 没回灌。
3. **安全靠闭环**：最小权限 + 动作分级 + 人在回路 + 沙箱回滚 + 审计，而非单点权限控制。
4. **记忆要治理**：可判断、可归一、可衰减、可审计，并用指标防污染。
5. **工程化 > 概念**：状态分层、评估体系、上下文/缓存优化，是真正的落地分水岭。
